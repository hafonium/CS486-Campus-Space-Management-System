# Database Design Validation

## Purpose
Cross-validate three artifacts (BR analysis, ERD, logical design) for structural alignment and internal consistency. Output a minimal error report — only actual mismatches, no pass narratives.

## Core Rule: No Inference
Use **only** explicit text in the three current output files as evidence. No guessing, no renaming, no domain knowledge, no memory. When uncertain, report it as a finding.

## Inputs
1. Business Requirements Analysis output
2. Conceptual ERD output
3. Logical Database Design output

## Workflow
Before each step, read the corresponding reference file in `references/` for the full detailed checklist and rules.

### Step 1: Structural Mapping & Internal Consistency
Read `references/step-1-source-truth-and-structural-scope.md`.

**Cross-document (ERD → Schema):**
- Every ERD entity → corresponding table in schema by exact name.
- Every ERD attribute → corresponding column in schema by exact name. Do not infer renames.
- Every ERD relationship → corresponding FK (1:1, 1:N) or junction table (M:N) in schema.
- Multi-role relationships → distinct FK column names per role.

**Internal consistency (within a single document):**
- ERD diagram cardinality vs. Relationship Summary table text.
- Logical design enum values vs. same document's prose references.
- Logical design FK summary table vs. schema FK declarations/nullability.
- Logical design NOT NULL summary table vs. schema column constraints.

**FAIL IF:** any entity/attribute/relationship has no schema counterpart; any internal contradiction found.

### Step 2: Keys & Relationships
Read `references/step-2-keys-and-relationships.md`.

- Each table has exactly one PK matching the PK summary table.
- Each FK is on the child/right side and aligns with ERD relationship cardinality.
- FK nullability in schema matches FK summary table and ERD cardinality.
- Each candidate key in summary table has a UNIQUE constraint in schema.
- Multi-role FKs: N distinct named columns for N relationship lines to same parent.

**FAIL IF:** PK mismatch, FK misplacement/nullability conflict, missing CK/UNIQUE, naming conflict.

### Step 3: Domains & Constraints
Read `references/step-3-domains-and-constraints.md`.

- Enum/domain values match across BR, ERD, and schema by exact spelling.
- Column data types are valid refinements of ERD abstract types (see equivalence table).
- NOT NULL annotations match the NOT NULL summary table.
- Default values match business-process defaults in logical design prose.
- Every CHECK constraint in logical design prose exists in schema.

**FAIL IF:** value spelling mismatch, invalid type, NOT NULL/default conflict, missing CHECK.

### Step 4: Business Rules
Read `references/step-4-business-rule-classification.md`.

**Classify each BR first:**
- **Database-Enforceable:** data integrity, uniqueness, referential integrity, required fields, value ranges, format constraints. Must appear as schema constraint (Structural) or in Procedural Enforcement.
- **Application-Layer:** external authentication (SSO/LDAP), user authorization logic, UI validation, workflow logic. Out of scope — do **not** flag.

For each Database-Enforceable rule:
- If Structural → confirm corresponding CHECK/constraint in schema.
- If Procedural → confirm documented in Procedural Enforcement section with enforcement strategy.
- If missing from both → report as gap.

**FAIL IF:** a Database-Enforceable rule has no structural constraint and no procedural documentation.

### Step 5: Report
Read `references/step-5-reporting-and-anti-hallucination.md`.

- **PASSED:** write exactly `No errors found.`
- **FAILED:** fill the findings table. One row per issue. No narrative outside table.

| Category | Description of Mismatch | Evidence (Quote Both Sides) | Recommended Fix |
| :--- | :--- | :--- | :--- |
| [category] | [what conflicts] | [source A: "..."; source B: "..."] | [change and where] |

**Categories:** `Structural Mapping` | `Key / Relationship` | `Domain Constraint` | `Business Rule Gap` | `Internal Inconsistency`

**Evidence rule:** quote both conflicting values exactly. Do not infer which is correct.
**Output style:** do not summarize correct sections, do not add observation notes, do not add separate review checklists unless explicitly asked.

## Data Type Equivalence
Only flag fundamental domain mismatches (e.g., ERD `int` → schema `varchar`). These refinements are valid:

| ERD abstract type | Valid schema concrete types |
| :--- | :--- |
| `string` | `varchar`, `text`, `char` |
| `int` | `integer`, `smallint`, `bigint`, `serial` |
| `datetime` | `timestamp`, `timestamptz`, `datetime` |

## Glossary
- **PK (Primary Key):** column(s) that uniquely identify each row. Every table has exactly one.
- **FK (Foreign Key):** column referencing another table's PK. Enforces referential integrity.
- **CK (Candidate Key):** alternative unique column(s). Declared via UNIQUE constraint.
- **Junction Table:** bridge table resolving M:N relationships with two FKs.
- **CHECK:** schema-level rule constraining column values (e.g., `CHECK (age >= 0)`).
- **NOT NULL:** column must always contain a value.
- **Structural:** enforced by schema (PK, FK, UNIQUE, NOT NULL, CHECK).
- **Procedural:** enforced by application logic, triggers, or stored procedures.

# Database Design Validation

## Purpose
This skill helps the agent evaluate whether the Step 3 Logical Database Design correctly represents the Step 2 Conceptual ERD, satisfies the Step 1 Business Requirements, and uses appropriate keys, relationships, and constraints.

The final output must clearly report every inconsistency found, explain why it is a problem, and state what should be corrected.

## When to use
Use this skill when:
- the user provides a business requirement document, ERD, and relational schema,
- the user asks to validate a database design,
- the task is specifically designated as "Database Design Validation" or "Step 4".

Do not use this skill when:
- the user only wants ERD generation,
- the user only wants logical schema generation,
- the user wants SQL implementation or sample data.

## Inputs
The agent should expect:
- The completed Step 1 Business Requirements.
- The completed Step 2 Conceptual ERD.
- The completed Step 3 Logical Database Design.

## Workflow

### Step 1: Check structural mapping
- Verify every ERD entity appears as a table.
- Verify every ERD relationship appears in the relational schema.
- Verify every M:N relationship is resolved using an appropriate junction table.

### Step 2: Check keys and relationships
- Verify each table has a correct primary key.
- Verify foreign keys are placed on the correct side of the relationship.
- Verify candidate keys, unique constraints, and multi-role foreign keys are correctly defined.

### Step 3: Check domains and constraints
- Verify data types, NOT NULL rules, enums, defaults, and other declared constraints match the business rules.
- Verify that status values and role values are constrained consistently.

### Step 4: Check business-rule consistency
- Verify the schema supports rules that can be enforced structurally.
- Identify rules that require triggers, stored procedures, or application logic, such as overlapping bookings or restricted status transitions.

### Step 5: Report all issues
- List every inconsistency found.
- Group issues by category.
- For each issue, explain what is wrong and how the schema should be fixed.
- If no issues are found, state that the validation passed.

## Output template
When producing the Step 4 deliverable, write the result into `outputs/04-design-validation-G09.md` and use this exact structure.

Use this exact structure:

### 1. Validation Status
* Select only one:
	- **VALIDATION PASSED**: The schema is fully consistent with the ERD and business rules.
	- **VALIDATION FAILED**: Inconsistencies detected. Action required.

### 2. Findings & Resolution Plan
* If passed, write: `No errors found.`
* If failed, list every issue in a table.

| Category | Description of Inconsistency | Source of Truth (ERD/Req) | Recommended Fix for Schema |
| :--- | :--- | :--- | :--- |
| [Category] | [Problem] | [ERD or Requirement] | [Fix] |

### 3. Action Required
* State the exact next step, such as returning to Step 3 to apply the recommended fixes.

## Review checklist
Before finishing, verify:
- all entities and relationships are represented correctly,
- keys and foreign keys are consistent,
- constraints and defaults match the business rules,
- complex business rules are identified clearly as procedural when needed,
- every inconsistency is listed explicitly in the findings table.

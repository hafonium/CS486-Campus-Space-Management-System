# Conceptual Database Design (ERD)

## Purpose
This skill helps the agent transform a structured Business Requirement Analysis (Step 1) into a visual and text-based Conceptual Database Design (Step 2). 

The final output must provide a clear, abstract map of the real-world domain, completely independent of relational database constraints. It prepares the user for Step 3 (Logical Schema Design).

## When to use
Use this skill when:
- the user provides a Business Requirement Analysis (actors, entities, relationships, rules).
- the user asks to draw, generate, or map a conceptual ERD.
- the task is specifically designated as "Conceptual Design" or "Step 2".

Do not use this skill when:
- the user asks for tables, primary/foreign keys, or SQL statements.
- the user wants to resolve Many-to-Many relationships into junction tables (that is Step 3).

## Inputs
The agent should expect:
- The completed Step 1 output (Entities, Attributes, Relationships, Cardinalities).
- Any specific formatting requests.

## Workflow

### Step 1: Map Entities and Attributes
- Extract the entities from the provided requirements.
- Identify the core attributes for each entity.
- Determine the logical Unique Identifier (Primary Key) for each entity.
- Ensure absolutely no Foreign Keys are added to the attribute lists.

Before this step, read `references/conceptual-attribute-rules.md`.

### Step 2: Map Relationships and Cardinality
- Translate the text-based business relationships into conceptual relationships.
- Determine cardinalities (1:1, 1:N, M:N).
- Add descriptive verb phrases to the relationships (e.g., "submits", "contains").
- Preserve all Many-to-Many relationships; do not resolve them.

Before this step, read `references/conceptual-relationships-guide.md`.

### Step 3: Generate the Mermaid Block
- Write the `mermaid` code block starting with `erDiagram`.
- List relationships first, then entity blocks below.
- Ensure attribute data types are conceptual (e.g., string, int, datetime).

Before this step, read `references/mermaid-syntax-guide.md`.

### Step 4: Generate the Conceptual Data Dictionary
- Create a clean text-based list detailing the entities.
- Mark the unique identifiers explicitly with `(PK)`.
- Reiterate the relationship summary in plain text below the entities.

### Step 5: Write the Design Notes
- Create a bulleted list explaining the conceptual choices made during Step 2.
- Explicitly state that Foreign Keys from Step 1 were removed to maintain conceptual purity.
- Explicitly state if any Many-to-Many (M:N) relationships were preserved without junction tables.
- Briefly explain any complex multi-role relationships (e.g., if a User has multiple different relationships with a Booking).
- Explain any changes made to relationship names/verbs compared to Step 1 (e.g., refining generic verbs into active, descriptive phrases).

## Output template
Use this exact structure:

```mermaid

    %% Relationships go here
    
    %% Entities go here
``` 
### 2. Conceptual Data Dictionary

**Entities and Attributes:**

  * UniqueIdentifier (PK)
  * AttributeName

**Relationship Summary:**
* **[ENTITY_1]** [Cardinality] **[ENTITY_2]**: [Short description].

### 3. Design Notes
1. **Foreign Keys Removed:** [Explain that FKs from Step 1 were excluded...]
2. **M:N Relationships Preserved:** [Explain that M:N relationships were kept intact...]
3. **Relationship Verbs Refined:** [Explain why specific relationship names from Step 1 were changed to more descriptive/active verbs in this ERD...]
4. **[Other Notes]:** [Any other relevant architectural decisions...]

## Review checklist
Before finishing, read `references/erd-review-checklist.md` and verify:
- no Foreign Keys exist inside the entities.
- M:N relationships are intact.
- the Mermaid syntax is strictly valid.

# Logical Database Design

## Purpose
This skill defines the operational protocol for transforming a Conceptual ERD (Step 2) into a physical Relational Schema targeted specifically at **Microsoft SQL Server (T-SQL)**.

Because T-SQL does not natively support ANSI `ENUM` types, the agent must map all categorical domains to standard VARCHAR and enforce them mathematically via explicit single-row CHECK constraints. Furthermore, the output must achieve absolute bi-directional Abstract Syntax Tree (AST) traceability between the DBML schema definitions and the textual constraint summaries.

## When to use
Use this skill when:
- The user provides a Conceptual ERD and Step 1 Business Requirements Analysis.
- The user commands the generation of a relational schema, tables, foreign key mappings, or DBML script.
- The task is specifically designated as "Logical Design" or "Step 3".

Do not use this skill when:
- The task requires executing SQL DDL/DML scripts (Step 5).
- The user is modifying conceptual entity cardinalities or drawing Mermaid diagrams (Step 2).

## Inputs
The agent requires:
- Completed Step 2 Conceptual ERD.
- Completed Step 1 Business Requirements (for constraint extraction).

## Workflow

### Step 1: Base Entity & Physical Type Normalization
- Convert every strong and weak entity from the ERD into a concrete DBML `Table` definition.
- Map abstract conceptual data types to concrete T-SQL physical types (`integer`, `varchar`, `datetime`, `bit`, `decimal`, `text`).
- Explicitly attach `[pk]`, `[increment]`, and mandatory `[not null]` tags.
- **Zero-Enum Policy:** Categorical attributes MUST be mapped as standard `varchar(50)`. Never declare `Enum` blocks.

Before this step, silently read `references/relation-mapping-guide.md`.

### Step 2: Relational Anchor Injection & Multi-Role Disambiguation
- Ensure all tables physically declare the underlying scalar Foreign Key (FK) columns prior to defining relationships.
- **Multi-Role Disambiguation:** When an entity participates in multiple distinct operational workflows with the same target table (e.g., `USER` submitting, approving, and checking in a `BOOKING`), the FK columns MUST be named strictly after their functional operational role, not the target table's name.

Before this step, silently read `references/foreign-key-placement.md`.

### Step 3: Pure Associative Entity De-composition (M:N Resolution)
- Physically instantiate an associative junction table for every Many-to-Many relationship (e.g., `SPACE_FACILITY`).
- Declare a explicitly named composite Primary Key constraint derived directly from the intersecting foreign keys: `(parent_a_id, parent_b_id) [pk, name: 'pk_[table_name]']`.
- Apply `[delete: cascade]` strictly to junction references; operational base entities must default to logical `RESTRICT` (omit delete modifiers).

Before this step, silently read `references/junction-table-rules.md`.

### Step 4: AST Bijection & Traceability Note Mounting
- Identify Candidate Keys (CK) and mark them explicitly with the `[unique]` tag in the DBML block.
- **AST Traceability Rule:** To ensure automated validation parsers achieve bi-directional mapping, every column restricted by a Section 3 single-row check MUST explicitly mount an inline pointer note matching its mathematical predicate:
  - Categorical fields: `note: 'CHECK ([column] IN (...)) – Section 3'`
  - Scalar bounds: `note: 'CHECK ([column] > 0) – Section 3'`
  - Chronological bounds: `note: 'CHECK ([start] < [end]) – Section 3'`

* **Note String Quoting Convention for Embedded Single Quotes:** When a note string references SQL constraints that contain single-quoted literals (e.g., `IN ('a','b')`), wrap the entire note value in double quotes to prevent DBML parsing errors:
  `note: "CHECK ([column] IN ('value1','value2')) – Section 3"`.
  Alternatively, escape inner single quotes with backslashes: `note: 'CHECK ([column] IN (\'value1\',\'value2\')) – Section 3'`, but double-quote wrapping is preferred for readability.

Before this step, silently read `references/dbml-syntax-guide.md`.

### Step 5: Bottom-Script Referential Block Synthesis
- Declare all referential integrity statements globally at the absolute bottom of the script under the `// --- Relationships ---` header.
- Syntax must strictly follow: `Ref: CHILD.fk_col > PARENT.pk_col`. Inline reference parameter declarations inside table blocks are strictly prohibited.

### Step 6: Procedural Business Logic Extraction
- Cross-reference Step 1 Business Rules to extract multi-row, cross-table, or state-machine constraints that surpass single-row CHECK capabilities (e.g., overlapping booking prevention, maintenance availability gates, role-based authorization barriers). Document these explicitly in Section 4 as requiring application middleware or T-SQL Triggers.

Before this step, silently read `references/meta-pattern-extraction-guide.md`.

### Step 7: Strict Document Synchronization Audit
- Verify 100% absolute bijection: Every attribute marked `[unique]` in DBML must be documented as a Candidate Key in Section 2. Every column documented as mandatory in text must carry `[not null]` in DBML. Section 2 must correctly render all three structural tables (Primary/Candidate Keys, Foreign Keys, NOT NULL Constraints).

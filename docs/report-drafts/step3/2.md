# Logical Database Design

## Purpose
This skill helps the agent transform a Conceptual ERD (Step 2) into a structured Logical Database Design (Step 3) targeted specifically at **Microsoft SQL Server (T-SQL)**. 

The final output must provide a strict relational schema written in DBML ready for SQL implementation. Because T-SQL does not natively support `ENUM` types, all categorical domains must be mapped as VARCHAR and enforced mathematically via explicit single-row CHECK constraints. It must also document complex multi-row business constraints that require procedural enforcement.

## When to use
Use this skill when:
- the user provides a Conceptual ERD and Business Requirement Analysis.
- the user asks to generate a relational schema, tables, map foreign keys, or write DBML.
- the task is specifically designated as "Logical Design" or "Step 3".

Do not use this skill when:
- the user asks for SQL DDL/DML statements (that is Step 5).
- the user is still asking to identify basic entities or draw Mermaid diagrams.

## Inputs
The agent should expect:
- The completed Step 2 Conceptual ERD.
- The completed Step 1 Business Requirements (for constraint validation).

## Workflow

### Step 1: Map Strong Entities & Concrete Types
- Convert every strong entity from the ERD into a Relation (Table).
- Map conceptual data types to concrete T-SQL data types (`integer`, `varchar`, `datetime`, `decimal`, `bit`, etc.). 
- **CRITICAL:** Categorical attributes (status, type, role) MUST be mapped as standard `varchar(50)` (or contextual length). Never use `Enum`.

Before this step, read `references/relation-mapping-guide.md`.

### Step 2: Map 1:N and 1:1 Relationships (Defensive References)
- Place a Foreign Key (FK) on the "Many" side referencing the Primary Key of the "One" side.
- For relationships involving the same two entities, create distinct, role-descriptive FK names.
- Default strictly to `RESTRICT` (or `NO ACTION`) delete behaviors to preserve historical audit trails.

Before this step, read `references/foreign-key-placement.md`.

### Step 3: Resolve M:N Relationships (Pure Junctions)
- Create an Associative Relation (Junction Table) for every M:N relationship with a Composite PK derived from the participating foreign keys.
- Apply `[delete: cascade]` strictly to pure associative junction tables where child records hold no independent existential meaning outside their parents.

Before this step, read `references/junction-table-rules.md`.

### Step 4: Define Constraints, Candidate Keys, and Defaults
- Identify Candidate Keys (CK) and mark them explicitly with the `[unique]` tag in the DBML block.
- Mark mandatory attributes explicitly as `[not null]`. Composite PK members in junction tables must carry `[not null]`.
- Inject logical `default: '...'` values for attributes possessing a natural initial lifecycle state.

Before this step, read `references/candidate-key-constraints.md`.

### Step 5: Draft the Schema in DBML (Zero Enum Tolerance)
- Convert the logical design into strict DBML syntax.
- **Strict Reference Placement:** Declare all references globally at the absolute bottom of the script using `Ref: CHILD.fk > PARENT.pk` syntax. Inline reference tags are strictly forbidden.

Before this step, read `references/dbml-syntax-guide.md`.

### Step 6: Meta-Pattern Business Logic Extraction
Cross-reference Step 1 Business Rules to extract rules into 3 strictly defined Meta-Patterns:
1. **Single-Column Value Boundaries:** Extract mathematical constraints for individual scalar attributes (e.g., positive integers `qty > 0`, valid percentages). Document as SQL `CHECK` expressions in Section 3.
2. **Intra-Record Dependencies:** Extract conditional or chronological dependencies between attributes within the same row (e.g., `start_time < end_time`). Document as SQL `CHECK` expressions in Section 3.
3. **Multi-Row / Cross-Table Procedural Exclusions:** Identify complex business gates involving multiple rows or cross-entity state verification (e.g., double-allocation limits, overdraft limits). Document in Section 4 as requiring *Triggers, Stored Procedures, or Application Logic*.

Before this step, read `references/meta-pattern-extraction-guide.md`.

### Step 7: Documentation-to-Code Synchronization Audit
- Verify 100% absolute synchronization: Every attribute marked `[unique]` in DBML must be documented as a Candidate Key in text. Every column documented as mandatory in text must carry `[not null]` in DBML. The documentation table of Foreign Keys must exactly reflect the global `Ref:` statements.

# Logical Database Design

## Purpose
This skill helps the agent transform a Conceptual ERD (Step 2) into a structured Logical Database Design (Step 3) targeted specifically at **Microsoft SQL Server (T-SQL)**. 

The final output must provide a strict relational schema written in DBML ready for SQL implementation. Because T-SQL does not natively support `ENUM` types, all categorical domains must be mapped as standard VARCHAR and enforced mathematically via explicit single-row CHECK constraints. It must also document complex multi-row business constraints that require procedural enforcement.

## When to use

**Use this skill for Step 3 (Logical Design)**, specifically when:
- The user provides a Conceptual ERD (from Step 2) and Business Requirement Analysis (from Step 1)
- The user designates work as "Logical Design" or "Step 3"
- The user requests help with any **Phase 3.1–3.4** execution
- Generating a relational schema, DBML tables, foreign keys, candidate keys, or domain constraints
- Validating or debugging existing logical designs against T-SQL standards

**Do NOT use this skill when:**
- The user asks for SQL DDL/DML implementations (that is Step 5)
- The user is still identifying basic entities or drawing Mermaid diagrams (that is Step 2)
- The user is designing the physical database or indexes (that is Step 4 or 5)
- The task is database implementation or query writing

## Workflow: 4-Phase Granular Execution

This skill is organized into **four distinct phases** that must be executed sequentially within Step 3 (Logical Design). Each phase has explicit input requirements, scope boundaries, and staged artifacts.

---

### Phase 3.1: Static Base Normalization

**Purpose:** Transform strong entities from the ERD into relational base tables mapped to concrete T-SQL types.

**Inputs:** 
- `@02-erd-design-G09.md`
- `relation-mapping-guide.md`
- `dbml-syntax-guide.md`

**Scope of Work:**
1. Map every strong entity and weak entity from the ERD into a DBML table.
2. Assign single-column PKs (either natural like `space_code` or surrogate `id integer [pk, increment]`).
3. Map conceptual data types to concrete T-SQL: `varchar`, `integer`, `datetime`, `bit`, `text`.
4. Mark mandatory columns explicitly with `[not null]`.
5. Set logical default values via `default: '...'`.
6. **Categorical Handling:** Map Enums as standard `varchar(50)` without declaring any `Enum {}` blocks.

**Forbidden in Phase 3.1:**
-  Foreign Key declarations (`Ref:` statements)
-  Candidate Key marks (`[unique]` tags)
-  CHECK constraints (domain or scalar)
-  Multi-role FK columns (e.g., `decision_staff_id`, `check_in_staff_id`)

**Staged Artifact:** `docs/report-drafts/step3/4-output.md`

---

### Phase 3.2: Referential Integrity Injection

**Purpose:** Inject all Foreign Key columns, resolve M:N relationships via junction tables, and declare global referential semantics.

**Inputs:**
- `@docs/report-drafts/step3/4-output.md`
- `@02-erd-design-G09.md`
- `foreign-key-placement.md`
- `junction-table-rules.md`

**Scope of Work:**
1. **Inject FK Columns:** Add all necessary multi-role scalar FK columns to operational tables (e.g., `requester_id`, `decision_staff_id`, `check_in_staff_id` in BOOKING).
2. **Create Junction Tables:** For every M:N relationship, declare a pure Associative `Table` with:
   - All participating FK columns marked `[not null]`
   - A composite PK in an explicit `Indexes { (col1, col2) [pk, name: 'pk_junction_name'] }`
   - **No surrogate key**
3. **Declare Global References:** Aggregate all `Ref: CHILD.fk > PARENT.pk` statements in a dedicated `// --- Relationships ---` block at the bottom:
   - Standard operational tables (USER, SPACE, BOOKING, MAINTENANCE_RECORD) use **no delete modifier** (defaults to `RESTRICT`)
   - Pure junction tables use **`[delete: cascade]`**
4. **Synchronize Documentation:** Update Section 2 (Foreign Keys table) with complete FK inventory.

**Example – SPACE_FACILITY Junction:**
```dbml
Table SPACE_FACILITY {
  space_code varchar(50) [not null]
  facility_id integer [not null]

  Indexes {
    (space_code, facility_id) [pk, name: 'pk_space_facility']
  }
}

Ref: SPACE_FACILITY.space_code > SPACE.space_code [delete: cascade]
Ref: SPACE_FACILITY.facility_id > FACILITY.facility_id [delete: cascade]
```

**Forbidden in Phase 3.2:**
- Candidate Key marks (`[unique]` tags)
- CHECK constraints

**Staged Artifact:** `docs/report-drafts/step3/5-output.md`

---

### Phase 3.3: Candidate Keys & Domain CHECKs

**Purpose:** Identify and tag all candidate keys; extract and document all business-level domain constraints as T-SQL CHECK expressions.

**Inputs:**
- `@docs/report-drafts/step3/5-output.md`
- `@01-business-req-analysis-G09.md`
- `candidate-key-constraints.md`
- `meta-pattern-extraction-guide.md`

**Scope of Work:**
1. **Candidate Key Identification:** 
   - Scan each table for minimal unique identifiers (e.g., `email`, `phone_number` in USER; `facility_name` in FACILITY; composite `(building, floor, room_number)` in SPACE).
   - Tag single-attribute CKs inline: `email varchar(255) [not null, unique]`
   - Tag composite CKs in an `Indexes` block: `Indexes { (building, floor, room_number) [unique, name: 'uq_space_location'] }`
   - **Synchronize with Section 2:** Every `[unique]` tag must be documented as a Candidate Key.

2. **Domain CHECK Extraction:**
   - **Pattern 1 (Single-Column Checks):** Categorical domains and scalar boundaries.
     - Example: `CHECK ([role] IN ('student', 'lecturer', ...))` 
     - Example: `CHECK ([capacity] > 0)`
   - **Pattern 2 (Intra-Record Checks):** State-contingent nullability and chronological progression using Propositional Logic ($P \implies Q$).
     - Example: `CHECK ([requested_start_time] < [requested_end_time])`
     - Example: `CHECK ([status] <> 'completed' OR ([completion_time] IS NOT NULL AND [result_note] IS NOT NULL))`

3. **CHECK Compilation in Section 3:** Organize all constraints by table, provide explicit constraint names, and reference business requirements.

4. **AST Traceability:** Insert inline `note:` attributes into DBML columns that reference `CHECK (...)` constraints:
   - Example: `capacity integer [not null, note: 'CHECK ([capacity] > 0) – Section 3']`

**Forbidden in Phase 3.3:**
-  Multi-row / cross-table checks (those belong in Section 4, Phase 3.4)
-  Changes to the structural FK layout

**Output Synchronization Gate:**
Before outputting 3.3, execute the internal 6-point **logical-review-checklist.md** (Gates 1-6).

**Staged Artifact:** `docs/report-drafts/step3/6-output.md`

---

### Phase 3.4: Architectural Gate Verification

**Purpose:** Document procedural enforcement requirements (cross-table and multi-row business rules), state machine lifecycles, and design justifications. Finalize the Logical Design report.

**Inputs:**
- `@docs/report-drafts/step3/6-output.md`
- `@01-business-req-analysis-G09.md`
- `logical-review-checklist.md`

**Scope of Work:**
1. **Section 4: Architectural Assumptions**
   - Enumerate all **Pattern 3 (Multi-Row / Cross-Table)** rules that cannot be enforced via single-row CHECK constraints.
   - For each rule, specify:
     - **Rule Name** (e.g., "Overlapping Booking Prevention")
     - **Business Requirement Source** (e.g., "BR 3")
     - **Enforcement Strategy:** T-SQL Trigger, Application Logic, or Stored Procedure with pseudo-code
   - Document state machine lifecycles (e.g., booking_status transitions: `pending` → `approved|rejected|cancelled` → ...)

2. **Section 5: Design Notes**
   - Explain key architectural decisions:
     - M:N junction table construction and cascade rationale
     - Multi-role FK disambiguation in BOOKING and MAINTENANCE_RECORD
     - T-SQL Enum prohibition and CHECK-based domain enforcement
     - Candidate Key placements and natural vs. surrogate identifier choices
   - Cross-reference assumptions from BR analysis

3. **Final Validation Gate (6-Point Checklist):**
   - Gate 1: Zero Enum audit
   - Gate 2: Candidate Key bi-directional parity
   - Gate 3: Referential Delete Behavior parity
   - Gate 4: Mandatory Nullability alignment
   - Gate 5: Categorical Domain Check coverage
   - Gate 6: Scalar Boundary traceability

**Staged Artifact:** `docs/report-drafts/step3/7-output.md` **(Final Logical Design Deliverable)**

## Output Template & Synchronization

The final deliverable (Phase 3.4) must follow this 5-section structure:

### Section 1: Relational Schema (DBML)
*Accumulated from Phases 3.1 + 3.2 + 3.3*

- Copy complete DBML here (tables + indexes + global Ref declarations)
- Include all candidate key tags `[unique]`
- Include all inline notes pointing to Section 3 CHECKs
- Instruction to paste into [dbdiagram.io](https://dbdiagram.io/)

### Section 2: Constraints and Keys Summary
*Accumulated from Phases 3.1 + 3.2 + 3.3*

Subsection A: **Primary Keys & Candidate Keys**
- Bulleted list: `TABLE_NAME: PK = [attr], CK = [attr]`
- Must bijectively match all `[unique]` tags in DBML

Subsection B: **Foreign Keys & Referential Integrity** 
- Table format: FK Column, Child Table, Parent Table, PK Column, Nullable, Delete Behavior
- Must bijectively match all `Ref:` statements in DBML

Subsection C: **NOT NULL Constraints** *(Optional summary)*
- Table format: Table Name, NOT NULL Columns

### Section 3: Business Integrity Constraints (T-SQL Domain CHECKs)
*Generated in Phase 3.3*

Organized by table. Each CHECK includes:
- Constraint name (e.g., `chk_user_role_domain`)
- T-SQL syntax: `CHECK ([col] IN (...))`  or `CHECK ([col] > 0)` or Propositional Logic
- Reference to business requirement if applicable

**Do NOT include Pattern 3 (cross-table, multi-row) checks here.** Those belong in Section 4.

### Section 4: Architectural Assumptions
*Generated in Phase 3.4*

**Procedural Enforcement (Multi-row / Cross-table logic):**
1. **Rule Name**
   - Business Requirement source
   - Enforcement strategy: T-SQL Trigger, Application Logic, or Stored Procedure
   - Pseudo-code or implementation sketch

**Design Assumptions:**
1. Scope declarations (e.g., "No recurring bookings", "Single role per session")
2. Candidate Key justifications (composite location identifier, natural vs. surrogate)
3. Multi-role FK disambiguation explanation
4. State machine lifecycle definitions

### Section 5: Design Notes
*Generated in Phase 3.4*

1. **M:N Resolution:** Explain each junction table, cascade rationale
2. **Multiple Role Mapping:** Explain multi-role FK column naming strategy
3. **T-SQL Compliance:** Document Enum prohibition and CHECK-based domain enforcement
4. **Referential Integrity:** Explain RESTRICT vs. CASCADE decisions
5. **Lookup Table Decisions:** Justify CHECK-based enumerations vs. separate lookup tables
